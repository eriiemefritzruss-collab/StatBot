# StatBot 技术文档：面向 AI + 统计的自主数据分析 Agent


## 1. 引言

### 1.1 项目背景与动机

在传统的数据分析与统计建模流程中，用户通常需要在多个工具之间反复切换：先读取和理解数据，再检查变量类型、缺失值和异常值；随后根据研究问题选择统计方法，编写分析代码，解释统计检验或模型结果，最后整理为可提交的报告。这个流程对统计知识、编程能力和软件工具熟练度都有较高要求。

大语言模型的发展为数据分析自动化提供了新的可能。LLM 可以理解自然语言需求，生成代码，并用自然语言解释结果。但如果只让模型自由生成 Python 代码，系统会面临几个问题：方法选择不稳定、统计假设容易遗漏、上下文容易污染、执行失败时缺少自动修复能力，以及外部模型超时时整条分析链路容易中断。

因此，本项目选择“AI + 统计 / 数据分析 Agent”作为方向，尝试构建一个更贴近统计分析工作流的 Agent。它不是只回答问题的聊天机器人，而是一个可以上传数据、规划分析、调用工具、执行代码、记忆上下文并生成报告的统计分析系统。

### 1.2 问题定义与目标

问题定义：

> 给定一个用户上传的数据集和自然语言分析需求，设计并实现一个能够自主规划统计分析步骤、选择合适统计工具或生成代码、调用执行环境、维护会话上下文、输出结果解释并生成技术报告的 Agent 系统。

具体目标包括：

1. 构建一个可交互的统计分析 Agent 前端，使用户可以上传数据、提问、查看数据表、编辑代码、导出 Notebook 和生成报告。
2. 将常见统计分析方法封装为结构化内置工具，降低纯代码生成的不确定性。
3. 使用 Jupyter Kernel 作为代码执行环境，保存每轮执行记录，提升分析过程的可复现性。
4. 引入会话级记忆，使 Agent 能记住上传数据、近期请求、工具选择、执行历史和生成产物。
5. 引入浏览器会话隔离，避免多个用户或多个浏览器会话之间共享错误状态。
6. 在外部 LLM 接口超时或不可用时提供本地降级路径，使分析执行结果、摘要和报告仍可生成。

## 2. 相关工作

### 2.1 大模型 Agent 与工具调用

大模型 Agent 通常由语言理解、任务规划、工具调用、环境观察和迭代决策等部分组成。典型 Agent 工作流强调“思考 -> 行动 -> 观察 -> 再决策”的闭环：
- 1.模型先理解用户目标
- 2.选择工具或生成动作
- 3.观察工具返回结果
- 4.根据结果继续推进任务。

在本项目中，工具调用并不仅指传统意义上的 LLM function calling，而是指 Agent 对外部计算环境和分析能力的协调，包括文件上传、数据读取、Jupyter Kernel 代码执行、图表生成、Notebook 导出和 Markdown 报告生成等。这样的设计让系统具备了真实的数据分析行动能力，而不是只停留在文本层面的问答。

### 2.2 自然语言驱动的数据分析

自然语言驱动的数据分析系统通常把用户请求转化为 SQL、Python、R 或可视化操作。此类系统的优势在于降低数据分析门槛，但难点在于：用户需求往往模糊，数据字段具有上下文含义，统计方法选择需要考虑变量类型、样本量、独立性、分布假设和研究目的。

如果系统仅依赖大模型自由生成代码，则容易出现以下风险：

- 选错统计方法，例如把配对样本当作独立样本处理；
- 忽略必要的假设检查，例如回归残差诊断或方差齐性检查；
- 生成无法运行的代码，或使用环境中不存在的库；
- 每轮对话之间状态不清晰，导致重复读取数据或引用错误变量。

因此，将高频统计方法封装为结构化工具，并结合会话记忆和执行环境，是提高系统稳定性的有效方式。

### 2.3 本项目的改进点

相较于原始 `LAMBDA` 工作流和一般的自由代码生成式数据 Agent，`StatBot` 的改进主要体现在以下方面：

1. **统计优先的系统定位**  
   系统默认启用 `analysis_profile: statistics`，在 prompt、工具库和报告模板中强调统计任务识别、假设检查、效应量、置信区间、模型诊断和结果解释。

2. **结构化统计工具注册表**  
   `builtin_skills.py` 中定义了 `AnalysisToolSpec` 和 `TOOL_REGISTRY`，将常见分析任务映射为可解释、可执行的内置代码模板。这样可以减少简单任务完全依赖 LLM 自由生成代码的风险。

3. **显式统计规划**  
   `conversation.py` 中的 `_build_statistical_plan()` 会在代码生成前构造统计分析计划，覆盖数据概览、任务类型、方法选择、假设诊断、输出指标和后续解释。

4. **会话级记忆与持久化**  
   `SessionState` 会记录上传文件、数据摘要、近期请求、统计计划、工具历史、执行摘要、错误信息和产物列表，使 Agent 具备跨轮对话的记忆能力。

5. **前端会话隔离**  
   `SessionManager` 以 Gradio 的 `request.session_hash` 为键，为每个浏览器会话创建独立 `StatBot` 实例，避免一个会话的代码或数据被另一个会话读取。

6. **模型超时降级机制**  
   当外部 LLM 在结果解释或报告生成阶段超时时，系统不会丢弃已经成功执行的结果，而是通过本地 fallback summary 和 fallback report 保持基本可用性。

## 3. 方法设计

### 3.1 总体架构

`StatBot` 采用分层式架构，由前端交互层、会话管理层、Agent 编排层、统计工具层、代码执行层、记忆层和产物输出层组成。

```text
User Query / Uploaded Data
          |
          v
  Gradio Frontend (statbot_app.py)
          |
          v
  SessionManager
          |
          v
  Per-session StatBot Instance
          |
          v
  Conversation Orchestrator
          |
          +--------------------------+
          |                          |
          v                          v
 Statistical Planner            SessionState
          |                          ^
          v                          |
 Built-in Tool Router --------------+
          |
          +--------------------------+
          |                          |
          v                          v
 Structured Statistical Code   Dynamic Code Generation
          |                          |
          +------------+-------------+
                       v
              Jupyter Kernel Executor
                       |
                       v
     Execution Results / Figures / Notebook / Report
```

系统的核心运行链路如下：

1. 用户通过 Gradio 前端上传数据或输入自然语言请求。
2. `SessionManager` 根据当前浏览器会话 ID 找到对应的 `StatBot` 实例。
3. `StatBot` 将请求交给 `Conversation` 编排器。
4. `Conversation` 先记录请求并生成统计分析计划。
5. 系统尝试通过 `route_builtin_skill()` 匹配内置统计工具。
6. 若匹配成功，系统注入工具代码；若未匹配，则调用 Programmer 模块生成动态 Python 代码。
7. `CodeKernel` 在 Jupyter Kernel 中执行代码并捕获文本、图片和错误信息。
8. 成功执行后，系统记录执行摘要和产物；若执行失败，则调用 Inspector 模块尝试修复。
9. 系统调用 LLM 解释结果；若 LLM 超时，则生成本地降级摘要。
10. 用户可继续提问、编辑代码、导出 Notebook 或生成报告。

### 3.2 Agent 架构设计

本项目内部包含三个主要智能角色模块：

| 模块 | 文件 | 主要职责 |
|---|---|---|
| Conversation | `conversation.py` | 负责整体任务编排、统计规划、工具路由、代码执行、错误修复、结果解释和报告生成 |
| Programmer | `programmer.py` | 负责调用大模型生成 Python 代码，支持 streaming 输出和可选知识注入 |
| Inspector | `inspector.py` | 在代码执行失败时，根据错误信息给出修复建议 |

这三个模块形成一个有限循环式 Agent：

1. `Conversation` 识别当前任务；
2. `Programmer` 或内置工具生成代码；
3. `CodeKernel` 执行代码；
4. 若失败，`Inspector` 诊断错误；
5. `Programmer` 根据诊断结果修复代码；
6. 循环最多执行 `max_attempts` 次；
7. 成功后进入结果解释与产物记录阶段。

这种设计既保留了大模型的开放任务处理能力，又用结构化工具和有限修复循环约束了执行过程。

### 3.3 核心算法与技术选择

#### 3.3.1 统计工具路由算法

`builtin_skills.py` 中定义了两个核心数据结构：

- `AnalysisContext`：保存用户请求、标准化文本、数据列名、数值列、类别列、命中列和时间列等上下文。
- `AnalysisToolSpec`：描述一个工具的名称、类别、功能说明、匹配器和代码构造器。

工具路由过程为：

```text
user_message + dataframe
        |
        v
build AnalysisContext
        |
        v
iterate TOOL_REGISTRY
        |
        v
matcher(ctx) == True ?
        |
        v
builder(ctx, spec) generates executable code
        |
        v
embed_skill_code(skill)
```

该算法目前以关键词匹配和字段类型推断为主。虽然语义理解能力不如大型检索器或分类器，但优点是可解释、可控、实现简单，适合课程项目和高频统计任务。

#### 3.3.2 统计规划

当 `statistical_planning` 打开时，系统会使用 `STAT_PLANNER_PROMPT` 生成六步统计计划：

1. 数据概览检查；
2. 研究问题 / 任务类型识别；
3. 统计方法选择；
4. 假设与诊断检查；
5. 推断 / 建模输出；
6. 解释与下一步。

该计划随后会通过 `STAT_PLAN_INJECTION` 注入给代码生成链路。这样做的目的不是让计划完全替代工具路由，而是给后续代码生成提供统计结构约束。

#### 3.3.3 代码执行与结果捕获

系统选择 Jupyter Kernel 作为执行环境，原因有三点：

1. Jupyter 天然适合数据分析和 Notebook 导出；
2. Kernel 可以跨轮保存变量，例如上传数据后变量 `data` 可被后续请求复用；
3. Kernel 的 IOPub 消息可以捕获标准输出、表格、图片和错误 traceback。

`kernel.py` 中的 `CodeKernel.execute_code()` 会将执行输出分为三类：

- 文本输出：写入对话展示区和 Notebook；
- 图片输出：保存为 PNG/JPEG 并登记为分析产物；
- 错误输出：清理 ANSI 控制字符后交给修复链路。

#### 3.3.4 上下文记忆

`SessionState` 使用 JSON 文件持久化会话状态。其记忆内容包括：

- 上传文件列表；
- 数据集行列数、字段名、缺失值；
- 最近用户请求；
- 最近完成任务摘要；
- 当前阶段；
- 最近统计计划；
- 最近成功代码；
- 最近执行摘要；
- 最近错误；
- 工具选择历史；
- 产物列表。

这些内容会被 `build_memory_context()` 汇总后注入后续请求，使 Agent 能在多轮交互中保持上下文连续。

### 3.4 工具与环境设计

#### 3.4.1 内置统计工具范围

当前 `TOOL_REGISTRY` 覆盖以下类别：

| 类别 | 工具示例 |
|---|---|
| 描述统计与数据质量 | 数据概览、描述统计、缺失值分析、异常值筛查、分组统计、频数统计、列联表 |
| 假设检验 | 单样本 t 检验、双样本 t 检验、配对 t 检验、Wilcoxon、Mann-Whitney U、ANOVA、Kruskal-Wallis、卡方、Fisher、双比例检验 |
| 关系分析 | Pearson/Spearman 相关矩阵、散点回归图 |
| 回归建模 | OLS 线性回归、Logistic 回归 |
| 可视化 | 分布图、时间趋势图、箱线图、Pairplot |

#### 3.4.2 前端工具

Gradio 前端提供以下功能：

- `Upload Data`：上传 CSV / Excel 数据；
- `Submit`：提交自然语言问题；
- `Show/Update DataFrame`：查看当前数据表；
- `Edit Code`：查看并编辑最近一次生成的 Python 代码；
- `Submit Code`：执行用户手动修改后的代码；
- `Notebook`：导出执行历史为 `.ipynb`；
- `Generate Report`：生成 Markdown 分析报告；
- `Save Dialogue`：保存对话和配置；
- `Clear All`：清空当前会话并重启 Kernel。

#### 3.4.3 配置与依赖

项目使用 `config.yaml` 管理模型和系统配置。默认模型为 OpenAI-compatible API 形式的 `qwen-max`，并通过环境变量 `${OPENAI_API_KEY}` 读取密钥。主要依赖包括 Gradio、OpenAI Python SDK、pandas、scipy、statsmodels、scikit-learn、matplotlib/seaborn、Jupyter Client、IPython Kernel 和 nbformat。

## 4. 实现细节

### 4.1 关键模块实现说明

#### 4.1.1 `statbot_app.py`：前端入口

`statbot_app.py` 使用 `gr.Blocks` 构建 Web UI，并将每个按钮绑定到后端函数。它的关键设计是：所有回调都不直接访问全局 `StatBot`，而是通过 `get_bot(request)` 按 `request.session_hash` 获取当前会话对应的实例。

此外，文件中还包含两个工程适配点：

1. `patch_starlette_template_response()` 用于兼容 Gradio 4.44.x 与较新 Starlette 模板参数形式差异；
2. `find_available_port()` 会在默认端口被占用时自动寻找后续可用端口。

#### 4.1.2 `statbot.py`：单会话 Agent 封装

`StatBot` 类负责读取配置、加载 `.env`、初始化缓存目录、创建 `Conversation` 实例，并提供前端按钮所需的高层方法。上传数据时，它会：

1. 解析 Gradio 文件对象；
2. 将文件复制到当前会话缓存目录；
3. 对 CSV / Excel 调用 `Conversation.add_data()`；
4. 将数据摘要写入 `SessionState`；
5. 将数据基本信息追加到 Programmer 的系统提示中。

这保证后续分析请求能够直接使用已经加载到 Kernel 中的 `data` 变量。

#### 4.1.3 `conversation.py`：Agent 核心编排器

`Conversation` 是项目最核心的模块。其 `stream_workflow()` 方法实现完整 Agent 工作流：

1. 接收用户请求；
2. 更新会话阶段；
3. 生成统计计划；
4. 将记忆上下文注入请求；
5. 路由内置统计工具或调用 Programmer 生成代码；
6. 提取 Python 代码块；
7. 执行代码；
8. 成功后记录执行结果并解释；
9. 失败后调用 Inspector 诊断并尝试修复；
10. 捕获异常并记录错误状态。

该模块还实现了两个重要降级方法：

- `_fallback_result_response()`：当结果解释阶段 LLM 超时时，基于执行输出生成本地摘要；
- `_build_local_report()`：当报告生成 LLM 超时时，基于 `SessionState` 和对话历史生成确定性的 Markdown 报告。

#### 4.1.4 `builtin_skills.py`：统计工具注册表

该文件是本项目相较原始数据 Agent 的核心增强。每个统计工具由匹配器和代码构造器组成。例如，相关性分析工具会识别 `"correlation"`、`"pearson"`、`"spearman"` 等关键词，并根据数据中的数值列生成相关矩阵和热力图代码；OLS 回归工具会选择目标列和特征列，使用 `statsmodels` 拟合模型，并输出 VIF 与残差诊断图。

工具对象包含以下字段：

- `name`：工具名；
- `category`：统计类别；
- `description`：功能说明；
- `rationale`：选择理由；
- `assumptions`：统计假设；
- `code`：可执行 Python 代码。

在前端输出中，系统会展示选中的工具名称、类别、说明、原因和假设，使用户能够理解 Agent 为什么选择该方法。

#### 4.1.5 `kernel.py`：代码执行与 Notebook 导出

`CodeKernel` 使用 `jupyter_client.KernelManager` 启动指定 `IPYKERNEL`。执行代码时，它监听 Kernel 的 IOPub 消息，并将结果转换为前端可展示内容和 Notebook cell 输出。

Notebook 导出由 `write_to_notebook()` 完成。每次执行的代码和输出都会追加到 `self.nb`，因此用户可以在分析结束后下载一个包含执行历史的 `notebook.ipynb`。

#### 4.1.6 `session_state.py`：会话记忆

`SessionState` 将会话记忆保存为 `session_state.json`。每次上传数据、生成计划、选择工具、执行代码、生成产物或发生错误时，系统都会调用对应的 `register_*` 方法更新状态。

这种设计使 Agent 能在长对话中回答“刚才用了什么方法”“上次结果是什么”“当前数据有哪些列”等上下文问题，也为本地报告生成提供了结构化依据。

#### 4.1.7 `session_manager.py`：会话隔离

原始单例式后端容易导致多浏览器会话共享一个 `StatBot` 实例。`SessionManager` 使用 `OrderedDict` 维护会话到 Agent 实例的映射，并在超过 `max_sessions` 时按 LRU 方式淘汰旧会话，同时关闭对应 Kernel。该模块解决了 `Edit Code` 可能读取其他会话历史代码的问题。

#### 4.1.8 `programmer.py` 与 `inspector.py`

`Programmer` 负责调用 LLM 生成代码，支持 streaming 输出，并在 `retrieval=True` 时调用 `knw_in.py` 做知识片段注入。`Inspector` 则在执行失败时读取错误代码和 traceback，给出修复建议。二者配合构成有限自修复机制。

#### 4.1.9 `knowledge_integration/`：可选知识注入

项目保留了可选知识注入模块，包含最近相关矩阵、统计检验助手、回归诊断、模式挖掘和神经网络示例等知识对象。默认配置中 `retrieval: False`，因此它不是当前演示链路的必需部分，但为后续扩展成 RAG 式统计 Agent 提供了基础。

### 4.2 技术难点及解决方案

| 技术难点 | 具体表现 | 解决方案 |
|---|---|---|
| 自由代码生成不稳定 | LLM 可能选错统计方法或生成不稳定代码 | 用 `builtin_skills.py` 将高频统计任务模板化 |
| 统计任务需要显式规划 | 用户需求往往模糊，直接写代码容易漏掉假设检查 | 在执行前生成六步统计分析计划 |
| 会话状态污染 | 多个浏览器会话共享同一 Agent 时，数据和代码可能串扰 | 用 `SessionManager` 基于 `session_hash` 创建独立实例 |
| 上传文件对象格式不统一 | Gradio 可能传入字符串、文件对象或列表 | 在 `_resolve_uploaded_file_path()` 中统一解析 |
| 模型解释超时 | 代码执行成功但最后解释失败，会让用户误以为任务失败 | 在 `_handle_execution_result()` 中加入本地 fallback summary |
| 报告生成依赖外部模型 | LLM 不可用时无法生成报告 | 在 `document_generation()` 中加入 `_build_local_report()` |
| 结果可复现性 | 对话式分析容易只留下文本，没有代码历史 | 将执行代码和输出写入 Notebook |
| 产物管理 | 代码执行可能生成图片、报告和其他文件 | `check_folder()` 扫描缓存目录并登记 artifact |

### 4.3 Agent 核心能力对应关系

| 课程要求 | StatBot 实现方式 |
|---|---|
| 自主决策与规划能力 | 使用统计规划 prompt，结合数据元信息选择工具或动态代码路径 |
| 工具调用与环境交互能力 | 调用文件系统、Jupyter Kernel、DataFrame 展示、Notebook 导出和报告生成 |
| 记忆与上下文管理能力 | 使用 `SessionState` 记录数据、请求、工具、执行、产物和错误 |
| 多步骤任务执行能力 | 完成“接收请求 -> 规划 -> 路由 -> 执行 -> 修复 -> 解释 -> 生成报告”的完整链路 |

### 5.4 案例展示：电商数据分析

以下为基于样例电商数据的典型交互流程。

#### 案例 1：数据概览

用户请求：

```text
Please read this dataset and tell me how many rows and columns it has.
```

系统行为：

1. 上传 `ecommerce_demo.csv` 后，系统读取数据并注册摘要；
2. 工具路由命中 `dataset_overview`；
3. 生成数据概览代码；
4. 输出行列数、重复行、字段类型、缺失率、数值摘要、类别摘要和前几行数据。

预期结果（真实结果见演示视频）：

- 数据集共有 48 行、17 列；
- 字段覆盖订单、地区、渠道、商品、价格、销量、收入、成本、利润、折扣、评分、配送天数和退货标记；
- 后续统计任务可以直接复用变量 `data`。

#### 案例 2：相关性分析

用户请求：

```text
Run a correlation analysis among numeric variables.
```

系统行为：

1. 工具路由命中 `correlation_matrix`；
2. 自动选择数值列；
3. 计算 Pearson 或 Spearman 相关矩阵；
4. 使用 seaborn 生成相关热力图；
5. 将图像保存到会话缓存目录并登记为 artifact。

该案例体现了 Agent 的“工具选择 + 执行环境交互 + 产物生成”能力。

#### 案例 3：OLS 回归分析

用户请求：

```text
Fit an OLS regression model for profit.
```

系统行为：

1. 工具路由命中 `ols_regression`；
2. 选择 `profit` 作为目标变量，使用其他数值列作为候选解释变量；
3. 使用 `statsmodels.OLS` 拟合线性回归模型；
4. 输出模型摘要、VIF 表、残差-拟合值图和残差分布图；
5. 结果解释 prompt 要求说明系数、诊断、置信区间、p 值和后续稳健性检查。

该案例体现了项目对“统计优先”的要求：不仅生成模型，还包含多重共线性和残差诊断。

#### 案例 4：模型超时降级

如果代码执行成功，但外部模型在结果解释阶段超时，系统会返回本地摘要，结构包括：

```text
Local fallback summary
Observed output preview: ...
Technical note: ...
Next, you can:
[1] Review the execution results above...
[2] Retry the interpretation step...
[3] Use Edit Code...
```

这种设计避免了“执行成功但最终显示失败”的用户体验问题，也体现了系统鲁棒性。

### 5.5 结果分析与讨论

实验和代码分析表明，`StatBot` 已经形成较完整的统计分析 Agent 闭环：

1. **任务链路完整**  
   系统能够从上传数据开始，完成规划、工具选择、代码执行、结果展示、产物保存和报告导出。

2. **统计工具提升稳定性**  
   对于数据概览、描述统计、相关分析、假设检验和回归等高频任务，结构化工具比自由生成代码更可控。

3. **会话记忆增强连续性**  
   通过 `SessionState`，Agent 能在后续问题中知道当前数据集、最近请求和已执行分析。

4. **降级机制改善演示可靠性**  
   即使外部模型超时，系统仍能保留执行结果、输出本地摘要并生成本地报告，适合课堂演示。

5. **仍存在局限**  
   目前工具路由主要依赖英文关键词和字段名匹配，对中文自然语言、变量同义词和复杂研究设计支持有限；代码执行环境也尚未实现严格沙箱，适合本地可信场景，不适合直接开放给不可信用户。

## 6. 总结与展望

### 6.1 项目总结

本文档介绍了 `StatBot` 的设计与实现。该系统围绕“AI + 统计”方向，将大语言模型、统计工具模板、Jupyter 代码执行、会话记忆和 Gradio 前端整合为一个自主数据分析 Agent。项目较好地体现了课程要求的四类核心能力：

- 自主决策与规划：通过统计计划和工具路由决定分析路径；
- 工具调用与环境交互：调用文件系统、Kernel、Notebook、图像和报告产物；
- 记忆与上下文管理：记录数据摘要、请求历史、工具历史和执行摘要；
- 多步骤任务执行：支持从数据上传到报告生成的完整链路。

整体来看，`StatBot` 已从“聊天式数据分析助手”进一步发展为“统计分析工作台式 Agent 原型”。

### 6.2 局限性分析

当前系统仍有以下不足：

1. **工具路由语义能力有限**  
   目前主要依赖英文关键词，中文请求、复杂意图和变量别名可能无法准确命中工具。

2. **变量选择仍偏启发式**  
   部分工具会默认选择第一个数值列或类别列作为目标变量 / 分组变量，真实分析中需要更强的目标识别和用户确认机制。

3. **执行安全边界不足**  
   代码直接在本地 Jupyter Kernel 中运行，不适合部署给不可信用户执行任意代码。

4. **测试体系不足**  
   仓库尚未形成系统化测试目录，路由、执行、前端回调和降级逻辑缺少自动化回归测试。

5. **统计报告表达仍可增强**  
   本地 fallback report 更偏执行摘要，叙事性和学术表达不如模型生成报告。

6. **知识注入默认关闭**  
   `knowledge_integration/` 中已有多个知识模块，但当前默认配置没有启用，尚未充分融入主工作流。

### 6.3 未来改进方向

未来可以从以下方向继续优化：

1. **增强工具路由**  
   使用 embedding 检索、轻量分类器或 LLM function calling，将用户意图映射到工具，同时支持中文关键词和字段别名。

2. **加入参数澄清机制**  
   当目标变量、分组变量或统计设计不明确时，Agent 应先向用户提出澄清问题，而不是直接猜测。

3. **扩展统计方法**  
   加入时间序列建模、生存分析、因果推断、贝叶斯分析、多重比较校正和稳健回归等高级方法。

4. **强化安全执行环境**  
   使用容器、资源限制、文件访问白名单和执行超时中断机制，降低任意代码执行风险。

5. **完善自动化测试**  
   建立 `tests/` 目录，覆盖工具路由、代码模板语法、数据上传、会话隔离、Notebook 导出和 fallback 行为。

6. **改进报告与前端体验**  
   支持 PDF 导出、图表目录、分析产物管理面板，以及点选式统计分析参数配置。

7. **量化 Agent 效果**  
   构建 benchmark，包括任务成功率、工具命中准确率、代码执行成功率、统计方法正确率和报告完整性评分。
